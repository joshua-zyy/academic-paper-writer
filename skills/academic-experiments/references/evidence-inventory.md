# 证据盘点指南

## 证据优先级

按可靠性从高到低区分三类：

1. **`newly_run`** — 当前会话中执行并记录了命令与输出。正文中的定量结果优先来自此类。
2. **`preexisting_artifact`** — 仓库中已存在的 checkpoint、日志、CSV、图表、表格，经检查后确认来源和含义。若无法重跑，可使用此类但必须标注来源与限制。
3. **`user_claim`** — 仅来自用户口述或旧草稿叙述，尚未本地复核。只能写成待核验信息，不能当作最终结果。

## 盘点清单

在跑命令前，先检查以下资源是否存在：

| 类别 | 文件/目录 | 关注点 |
|------|----------|--------|
| 配置 | `config.*` | 超参数、数据路径、模型结构 |
| 入口 | `main.*` / `train.*` / `eval.*` | 训练/评估入口 |
| 脚本 | `run_*.sh` / `run_*.ps1` | 可复现的运行命令 |
| 环境 | `requirements.txt` / `environment.yml` | 依赖与环境 |
| 模型 | `checkpoints/` / `*.pth` / `*.ckpt` | 已有模型权重 |
| 输出 | `logs/` / `outputs/` / `results/` | 运行日志与结果 |
| 指标 | `*.csv` / `*.json` | 量化指标文件 |
| 图表 | `*.png` / `*.pdf` / `*.svg` | 可视化产物 |

## 盘点四个核心问题

完成盘点后，回答：

1. **能否只评估已有 checkpoint，而无需重训？**
   - 优先路线：加载已有 checkpoint → 运行评估脚本 → 记录指标

2. **已有结果是窗口级、样本级还是受试者级？**
   - 这决定了结论可推广的粒度

3. **当前 split 协议是否可靠？**
   - 检查：训练/验证/测试划分方式、是否存在泄漏

4. **哪些结果可以在合理时间内复核？**
   - 评估脚本通常几分钟；完整训练可能需要数小时或数天
